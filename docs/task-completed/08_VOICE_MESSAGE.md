# 语音消息需求分析与任务编排

## 背景

语音消息包含录音、波形生成、上传、消息展示和播放。桌面端依赖 Electron 音频服务和本地文件路径；Web 端需要使用浏览器 MediaRecorder / Web Audio，并尽量复用或扩展 `packages/audio-core`，避免在页面内散落音频算法。

语音消息发送依赖 [06 媒体上传](06_MEDIA_UPLOAD.md)，消息展示和状态流转依赖 [04 消息收发](04_MESSAGING.md)。

## 当前基础

### 已具备

- `packages/audio-core` 已有录音、波形、播放相关源码：`recorder`、`waveform`、`player`。
- 桌面端已有 `saveVoice`、`getAudioInfoByLocalPath` 的能力边界。
- 前端已有录音按钮、音频消息、波形绘制、全局播放状态等设计参考。
- 媒体上传模块可上传 Blob 并由服务端创建语音消息。
- `packages/chat_ui` 可提供 Button、Tooltip、Spinner 等基础 UI；播放进度和录音状态先以现有组件组合展示，不假设项目已经有额外进度条组件。

### 主要缺口

- Web 端需要确认 `audio-core` 是否已有浏览器录音实现，缺失时应扩展 `audio-core`，不要在页面私有代码里实现完整音频核心。
- Web 端需要处理浏览器录音格式差异。
- Web 端不保存本地音频路径，播放使用远端 URL 或 object URL。
- 缺少 waveform、duration、mimeType 与消息协议字段的映射确认。

## 目标

### 一期目标

- 用户可以在 Web 端录制语音。
- 录音后生成 duration、waveform、mimeType。
- 录音 Blob 走媒体上传链路。
- 上传完成后服务端创建语音消息。
- 用户可以播放语音消息。
- 同一时间只播放一条语音，播放进度和波形联动。

### 二期方向

- 录音降噪、增益、静音检测。
- 语音转文字。
- 统一音频转码。
- Web Worker 波形计算。
- 更多浏览器兼容测试矩阵。

## 产品需求

### 录音

用户通过聊天输入区的录音按钮开始录音，结束后可发送或取消。

验收标准：

- 点击或按住开始录音。
- 录音时显示时长。
- 支持取消录音。
- 支持发送录音。
- 麦克风权限拒绝时展示提示。
- 录音不足最小时长时自动取消。
- 超过最大时长时自动结束。

### 波形和元信息

录音结束后生成语音消息需要的元信息。

验收标准：

- 生成 `duration`。
- 生成 `waveform`。
- 生成 `mimeType`。
- waveform 采样点数稳定。
- metadata 结构和服务端/消息类型约定一致。
- 生成失败时展示错误并允许重录。

### 语音上传与发送

录音 Blob 通过媒体上传链路提交，由服务端创建语音消息。

验收标准：

- 录音 Blob 作为上传文件提交。
- 上传入参包含 `duration`、`waveform`、`mimeType`。
- 上传过程展示 pending / uploading 状态。
- complete 后客户端不二次 `sendMessage`。
- 服务端通过 `newUpdateMessage` 推送最终语音消息。
- 上传失败可重试或重新录制。

### 语音播放

点击语音消息播放，并与全局播放状态联动。

验收标准：

- 点击语音消息开始播放。
- 再次点击可暂停或重新播放。
- 同一时间只播放一条语音。
- 切换语音时停止上一条。
- 播放进度实时更新。
- 波形随播放进度高亮。
- 播放结束后状态复位。
- 播放失败时展示错误。

### 语音消息展示

语音消息在聊天记录中有清晰状态和可操作区域。

验收标准：

- 展示时长。
- 展示波形。
- 展示播放/暂停状态。
- 自己和他人的气泡样式区分。
- loading、uploading、fail 状态明确。
- 失败语音可重试上传或重录。

## Web 适配要点

- 音频核心能力优先放在 `packages/audio-core`，页面只组合 hook 和 UI。
- 录音使用 MediaRecorder，需做 MIME 支持检测。
- 波形生成优先复用 `audio-core`；如需 Web Audio 解码，也应沉淀到 `audio-core`。
- Web 不使用 `saveVoice` 保存本地文件，不使用 `getAudioInfoByLocalPath` 读取本地路径。
- 语音 Blob 上传走 [06 媒体上传](06_MEDIA_UPLOAD.md)。
- 播放使用远端 URL 或 object URL。
- UI 使用 `chat_ui` / shadcn，不手写基础按钮、浮层、进度反馈。

## 风险点

- MediaRecorder 支持格式因浏览器不同而不同，Safari 需要重点验证。
- 麦克风权限只能在 HTTPS 或 localhost 下可靠使用。
- 波形生成可能阻塞主线程，长语音后续可放到 Worker。
- 如果页面绕过 `audio-core` 自己实现音频算法，后续桌面端和 Web 端会分叉。
- 语音发送依赖上传链路，上传未完成前不能声称语音消息闭环完成。
