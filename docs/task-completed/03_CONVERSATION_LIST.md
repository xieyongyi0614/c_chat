# 会话列表需求分析与任务编排

## 背景

会话列表是 IM 主界面的入口，承载私聊、群聊、未读、最后一条消息、实时更新和本地缓存。桌面端已经有较完整的会话列表能力。Web 端需要复刻产品行为，但调用层走 `shared-api`，缓存层走 IndexedDB，UI 继续复用现有聊天页面模式和 `chat_ui` 基础组件。

## 当前基础

### 已具备

- `shared-types` 已有 `LocalConversationListItem`、会话类型、分页响应等类型。
- `shared-protobuf` 已有 `getConversationList`、`newConversation`、`newUpdateMessage` 等事件。
- 桌面端已有全部 / 未读 / 私聊 / 群组 / 归档筛选思路。
- 群聊基础已使用 `Conversation.type = 2`。
- 本地会话缓存可由 Web 端 IndexedDB 替代 Electron SQLite。

### 主要缺口

- Web 端需要在 `shared-api` 内补齐会话相关 service：拉取会话、本地回退、创建私聊、已读同步。
- Web 端需要定义 IndexedDB conversation store。
- Web 端需要复刻进入聊天页、页面可见、周期同步的会话刷新策略。
- 归档、手动置顶如果后端未完整支持，一期需明确只保留入口或不实现。

## 目标

### 一期目标

- 用户登录后能看到会话列表。
- 会话列表支持全部、未读、私聊、群组筛选。
- 会话项展示头像、名称、最后一条消息、时间、未读数。
- 收到新消息或新会话推送时列表 upsert 并更新排序。
- 进入会话后未读清零并同步服务端已读位置。
- 网络失败时可使用 IndexedDB 本地缓存回退。

### 二期方向

- 手动置顶、归档、免打扰。
- 多标签会话同步。
- 会话搜索。
- 会话级草稿。

## 产品需求

### 会话列表展示

会话列表展示私聊和群聊的基础信息。

验收标准：

- 私聊展示对端头像、昵称和最后一条消息。
- 群聊展示群头像、群名称和最后一条消息。
- 没有头像时使用 Avatar fallback。
- 最后一条消息文案复用统一生成逻辑：图片 `[图片]`、视频 `[视频]`、文件 `[文件]`、音频 `[音频]`。
- 空列表展示 empty 状态。
- loading 使用 Skeleton 或 Spinner。

### 会话筛选

用户可以按 folder 查看不同会话集合。

验收标准：

- 全部：展示所有正常会话。
- 未读：展示 `unreadCount > 0` 的会话。
- 私聊：展示 `type === 1` 的会话。
- 群组：展示 `type === 2` 的会话。
- 归档：一期如无后端支持，明确展示空状态或隐藏入口，不做假功能。

### 实时更新

收到服务端推送时，会话列表实时更新。

验收标准：

- `newConversation` 推送可以插入新会话。
- `newUpdateMessage` 推送可以更新会话最后消息、时间、未读数。
- 同一会话重复推送不会插入重复项。
- 当前选中会话收到新消息时，本地未读保持清零并触发已读同步。
- 未选中会话收到新消息时，未读数按服务端口径更新。

### 已读同步

进入会话或阅读到最新消息时，同步已读位置。

验收标准：

- 点击会话后本地立即清零未读。
- 防抖调用消息已读接口。
- 服务端返回 `lastReadSeq` 后更新本地会话。
- 已读失败时不阻塞用户查看，但需要可重试或下次同步修正。

### 服务端优先与本地回退

会话数据优先使用服务端结果，本地缓存用于快速展示和网络失败回退。

验收标准：

- 进入聊天页时先可展示本地缓存。
- 服务端返回后用服务端列表覆盖或 upsert 本地列表。
- 网络失败时保留本地缓存并展示离线状态。
- 过期回包不会覆盖当前用户或当前会话的数据。

### 周期同步

Web 端需要补齐离线期间的新会话、群变更和未读变化。

验收标准：

- 进入聊天页时同步一次。
- 页面从隐藏变为可见时同步一次。
- 页面可见时按固定周期同步一次。
- 同步策略不与实时推送冲突，upsert 保持幂等。

## Web 适配要点

- 会话接口应沉淀到 `shared-api`，页面不直接使用 protobuf event。
- IndexedDB conversation store 字段对齐 `LocalConversationListItem`。
- folder 筛选 UI 使用 `chat_ui` / shadcn 的按钮、Tabs 或 ToggleGroup，不手写基础交互。
- Avatar、Badge、ScrollArea、Skeleton、Tooltip 等优先使用 `chat_ui`。
- Web 一期单账号，不处理多窗口会话隔离。

## 风险点

- 未读数必须以服务端为准，不能纯客户端自增。
- 推送和周期同步可能乱序到达，需要 upsert 幂等。
- 本地缓存回退不能展示上一个账号的数据。
- 归档和手动置顶如果后端未支持，不要在 Web 端做假状态。
- IndexedDB 异步读取可能导致首屏闪烁，需要明确 loading 和本地缓存优先策略。
