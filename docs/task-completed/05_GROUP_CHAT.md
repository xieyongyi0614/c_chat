# 群聊需求分析与任务编排

## 背景

群聊完整后端和桌面端一期需求已记录在 [GROUP_CHAT_REQUIREMENTS.md](../requirements/GROUP_CHAT_REQUIREMENTS.md)。Web 端不重新设计群聊后端，也不复制一套协议，只复刻前端产品流程，并复用会话、消息、用户搜索、上传和共享协议能力。

## 当前基础

### 已具备

- 服务端已有群聊基础闭环，详见 [GROUP_CHAT_REQUIREMENTS.md](../requirements/GROUP_CHAT_REQUIREMENTS.md)。
- `shared-protobuf` 已有 `createGroup`、`getGroupDetail`、`updateGroup`、`inviteGroupMembers`、`leaveGroup`、`dismissGroup` 等群聊事件。
- `shared-types` 已有群聊 IPC/参数/返回相关类型。
- 会话列表已支持 `Conversation.type = 2` 群聊。
- 消息链路基于 `conversationId`，群消息可复用 [04 消息收发](04_MESSAGING.md)。
- 用户列表/搜索来自 [01 认证与用户](01_AUTH_USER.md)。

### 主要缺口

- Web 端需要在 `shared-api` 补齐群聊 service。
- Web 端需要复刻创建群 UI、群详情、成员列表、群资料编辑和成员管理。
- 群头像上传需要走 Web 媒体上传链路。
- 退出/解散/邀请后的会话列表同步需要和 [03 会话列表](03_CONVERSATION_LIST.md) 联动。

## 目标

### 一期目标

- 用户可以创建群聊并选择多个成员。
- 群聊出现在会话列表的群组筛选中。
- 群成员可以发送和接收群消息。
- 用户可以查看群资料和群成员。
- 群主可以编辑群名称、头像、公告。
- 群主可以邀请成员、解散群；普通成员可以退出群。
- 退出或解散后会话状态正确同步。

### 二期方向

- 管理员、踢人、禁言。
- @成员、@所有人。
- 群二维码、邀请链接、入群审批。
- 群公告已读、群消息已读人数。
- 群成员搜索。

## 产品需求

### 创建群聊

用户从聊天页进入创建群流程，选择至少 2 个其他用户，填写群名称或使用服务端默认名称。

验收标准：

- 选人能力复用用户列表/搜索。
- 至少选择 2 个其他用户。
- 群名称可空，由服务端生成默认名称。
- 群名称长度沿用桌面端限制。
- 可选群头像；头像上传后传 `avatarUrl`。
- 创建成功后会话列表 upsert 群会话并选中。

### 群会话展示

群聊作为独立会话展示。

验收标准：

- `type === 2` 展示群名和群头像。
- 群组筛选只展示群聊。
- 群头像为空时使用 Avatar fallback。
- 群消息到达后更新最后一条消息和未读数。
- 群会话历史拉取复用消息历史逻辑。

### 群消息

群成员可以在群会话内收发消息。

验收标准：

- 文本、图片、文件、视频、语音都复用现有消息链路。
- 非群成员不能发送群消息。
- 退出或被移除成员不能继续发送群消息。
- 解散群不能继续发送消息。
- 失败发送有明确错误状态。

### 群资料

用户可以查看群基础资料和成员列表。

验收标准：

- 展示群名称、头像、公告、群主、创建时间。
- 展示成员列表，包含头像、昵称、角色。
- 群主可看到编辑入口。
- 普通成员只读。
- loading、empty、error 状态完整。

### 成员管理

一期只做轻量管理。

验收标准：

- 群主可以邀请新成员。
- 普通成员可以退出群。
- 群主可以解散群。
- 群主退出一期按解散处理，不做群主转让。
- 邀请、退出、解散成功后刷新群详情和会话列表。
- 高影响操作使用 AlertDialog 确认。

## Web 适配要点

- 群聊业务调用应沉淀到 `shared-api` 群聊 service，不在页面拼 socket event。
- 群聊协议复用 `shared-protobuf`。
- 群聊类型复用 `shared-types`。
- 成员选择、群资料弹窗、编辑表单使用 `chat_ui` / shadcn 的 Dialog、Sheet、Command、Form、Input、Avatar、Button、AlertDialog。
- 群头像上传依赖 [06 媒体上传](06_MEDIA_UPLOAD.md)。
- 群消息依赖 [04 消息收发](04_MESSAGING.md)。
- 群会话同步依赖 [03 会话列表](03_CONVERSATION_LIST.md)。

## 风险点

- 群聊不要重写后端需求，后端边界以 [GROUP_CHAT_REQUIREMENTS.md](../requirements/GROUP_CHAT_REQUIREMENTS.md) 为准。
- 退出和解散必须同步会话列表，否则会出现不可用会话仍可发送的问题。
- 群消息未读数按 `ConversationParticipant` 维度维护，不能客户端自增。
- 群头像上传依赖媒体上传能力。
- 成员选择和邀请不能使用 mock 用户列表。
